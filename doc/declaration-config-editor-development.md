# 申报配置增强 — 开发文档（管理端 + 填报端 + 预览）

本文约定**管理端编辑**与**教师填报**并行推进时的目标、分层复用与迭代节奏，并与 [project-declaration-config-design.md](./project-declaration-config-design.md) 中的「项目 → 模块 → 子模块（map/list）」模型一致。

---

## 1. 目标

| 维度 | 说明 |
|------|------|
| 管理端 | 更易编辑：拖拽排序、复制模块/子模块、（后续）模板插入；**编辑时实时/按需预览**，避免「保存后才知道长什么样」。 |
| 填报端 | 同一套配置 JSON **驱动渲染**，与预览使用同一套「渲染管线」，减少两套 UI 分叉。 |
| 复用 | **类型定义、配置归一化、Map/List 渲染组件**在管理端预览与填报端共用；仅「是否可编辑、是否走保存接口」等用 props 区分。 |

---

## 2. 核心原则

1. **单一数据源**：已发布/草稿中的 `config`（`modules[]`）为唯一真相；预览与填报都基于同一份结构（填报可增加「当前用户草稿值」层，见下节）。
2. **预览即填报**：管理端预览区 = 填报端渲染组件 + `readOnly`（或 `preview`）模式，**禁止**再写一套仅用于预览的静态页面。
3. **先契约后功能**：`declarationConfigTransforms` 与后端校验形状保持一致；新增字段先改类型与文档，再改两端 UI。

---

## 3. 分层与复用（建议目录）

以下为逻辑分层，实际目录可按项目习惯微调。

| 层级 | 职责 | 复用方 |
|------|------|--------|
| **配置模型与校验** | `normalize` / `config ↔ 表单值` 转换（已有 `declarationConfigTransforms.ts`） | 管理端编辑器、预览、填报、保存前校验 |
| **只读/可编辑渲染** | 按 Module → SubModule 渲染 Map 字段、List 表格、附件占位 | **预览（只读）**、**填报（可编辑）** |
| **管理端专用** | 树形大纲、拖拽排序、复制、模板面板、与草稿版本 API 交互 | 仅管理端 |
| **填报专用** | 与「我的申报」草稿/提交、权限、项目开放状态绑定 | 仅填报端 |

建议将 **「配置渲染」** 抽到例如 `frontend/src/features/declaration-config-render/`（或 `components/declaration-config/`），管理端预览与填报页均从此引用。

**数据叠加约定**

- **预览**：`config` + 可选「示例/mock 值」或空值，只读。
- **填报**：`config` + **用户填报数据**（可与配置同构的 `values` 或后端约定的存储结构）；渲染层通过 props 绑定值与 `onChange`。

## 4. 并行开发：两条轨道

### 4.1 轨道 A — 管理端（编辑 + 预览）

| 阶段 | 内容 | 依赖 |
|------|------|------|
| A0 | 抽出 **DeclarationConfigPreview**（只读），数据来自当前编辑器中的 config（内存态） | 共享渲染层骨架 |
| A1 | 编辑器布局：左侧结构树或模块卡片 + **右侧预览**（分屏或 Tab「编辑 / 预览」） | A0 |
| A2 | **拖拽排序**（模块、子模块；必要时 list 行序另议）→ 写回 `order` 与数组顺序 | dnd 库 |
| A3 | **复制模块 / 复制子模块**（生成新 `key`、去重提示） | 无后端或仅前端 |
| A4 | （可选）**模块模板**：静态 JSON 片段一键插入 | 静态资源 |
| A5 | （后续）**服务端模板库**：保存/引用模板 API | 后端表设计 |

### 4.2 轨道 B — 填报端（配置驱动）

| 阶段 | 内容 | 依赖 |
|------|------|------|
| B0 | 拉取**当前项目已发布配置**（已有或扩展 API），归一化为渲染输入 | 与 design 文档一致 |
| B1 | 使用与预览 **相同的 Map/List 渲染组件**，接入真实草稿存储与校验 | 共享渲染层 |
| B2 | 列表行增删、附件上传、与后端字段对齐 | 依业务 API |
| B3 | 打印/导出（若需要）与只读视图复用同一渲染 | 可选 |

**并行方式**：优先完成 **共享渲染层（A0 + B0/B1 的基础组件）**；管理端先做预览，填报端接同一组件做可编辑，避免两条线长期分裂。

---

## 5. 功能迭代（与前期建议路线对齐）

| 迭代 | 管理端 | 填报端 | 复用点 |
|------|--------|--------|--------|
| **一期** | 预览面板（只读）+ 拖拽排序 + 复制模块/子模块 | 基于发布配置的填报主路径跑通（至少 Map + List 各一条链路） | 渲染组件 + transforms |
| **二期** | 静态模块模板插入、折叠与校验提示优化 | 完整附件、List 工具栏与项目规则一致 | 同左 |
| **三期** | 服务端模板库、权限与审计（若需要） | 性能与移动端适配、历史版本兼容策略 | 模板 JSON 契约 |

---

## 6. 管理端预览交互（建议）

- **触发**：切换「预览」Tab，或编辑区与预览区分屏实时刷新（注意大 JSON 时的节流）。
- **数据**：使用**当前未保存配置**（内存）预览时，标题旁提示「预览未保存内容」。
- **一致性**：预览区组件与填报页入口组件 **import 路径一致**（同一导出），样式可与全局面包屑区分（如加 `preview` 包裹 class）。

---

## 7. 后端与接口（概要）

| 项 | 说明 |
|----|------|
| 已发布配置拉取 | 填报端使用；需与 `project_id`、发布状态一致（见现有 declaration-config API）。 |
| 草稿编辑 | 管理端继续走现有草稿版本 CRUD；预览不强制新接口。 |
| 模板库（三期） | 新增「模板」资源 CRUD 或文件存储；与 `config.modules` 片段格式一致即可导入。 |

详细字段以 [project-declaration-config-design.md](./project-declaration-config-design.md) 为准。

---

## 8. 测试与验收

- **契约测试**：对样例 `config`（含 map/list、附件项）做 snapshot 或单测：归一化 ↔ 表单值 ↔ 再序列化形状稳定。
- **双端联调**：同一 `config` 在预览与填报只读模式下 DOM/结构一致（关键字段存在、顺序一致）。
- **回归**：管理端保存后，填报端拉取发布版本，字段与预览一致。

---

## 9. 风险与规避

| 风险 | 规避 |
|------|------|
| 预览与填报渲染分叉 | 强制共用 `features/declaration-config-render`，Code Review 禁止复制粘贴第二套 |
| 未保存预览与已发布不一致 | UI 明确标注；可选「对比已发布」仅二期 |
| 拖拽与表单受控状态冲突 | 排序只改 `modules[]` 顺序与 `order` 字段，再一次性 `setFieldsValue` 或更新 editor state |

---

## 10. 文档与实现同步

- 配置字段变更：先更新 **project-declaration-config-design.md**，再改 `declarationConfigTransforms` 与共享渲染。
- 本文件：**迭代目标与范围变更时更新阶段表**。
