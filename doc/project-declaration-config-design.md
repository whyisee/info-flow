# 申报项目配置设计

本文描述**申报项目配置**的领域模型与扩展方式，使同一套系统能通过配置适配**不同评选/申报项目**所需的材料结构、表单字段与附件要求，而不必为每个项目硬编码页面。

**与现有实现的关系**：当前系统中的「申报书填写」「基本信息」等页面是具体实现示例；本设计给出可演进的**配置层**抽象，便于后续将 UI 与存储迁移到「配置驱动」。

---

## 1. 背景与目标

### 1.1 问题

- 不同申报项目（如某类人才计划、教学成果奖）在**栏目划分、表格列、汇总句式、附件格式与大小**上差异大。
- 若每个项目单独开发页面，维护成本高、上线慢，且难以统一权限、草稿、提交与归档。

### 1.2 目标

1. **配置驱动**：用结构化配置描述「项目 → 模块 → 子模块」及每层的表单与附件，前端按配置渲染。
2. **子模块类型可组合**：同一模块下可同时存在**键值/汇总型（Map）**与**多行明细（List）**。
3. **可演进**：支持新增字段、调整列表列、模板文件与校验规则，并考虑**版本**与**兼容历史数据**。

---

## 2. 整体架构（概念）

```
项目（Project）
├── 模块 A（Module）
│   ├── 子模块 1（SubModule）
│   │   ├── 类型：map | list
│   │   ├── 填报：表单字段定义（含展示/校验）
│   │   └── 附件：上传项定义（模板、格式、大小、是否必填）
│   └── 子模块 2 …
├── 模块 B …
└── …
```

- **项目**：一次评选或申报活动的逻辑容器（如「某年度 XX 人才申报」），绑定一套**申报配置版本**。
- **模块**：一级业务分区，通常对应导航或 Tab（如「立德树人」下的「概述」「人才培养」「教学成果」）。
- **子模块**：模块内的最小可配置单元，决定**如何采集数据**：
  - **Map 类型**：一组命名字段或一句「句式填空」中的多个空位，适合概述、统计数据汇总。
  - **List 类型**：带行数上限的多行表（如「不超过 10 项」），每行可有独立附件列。

---

## 3. 子模块类型说明

### 3.1 Map 类型（键值 / 汇总）

**适用**：固定键集合、或「近五年，候选人……讲授 [ ] 门课程……」这类**嵌入式输入**的汇总说明。

**能力要点**：

- 字段列表：键、标签、控件类型（数字、文本、选择等）、校验（必填、范围、正则）。
- 可选：**一句模板文案**，其中用占位符引用字段，用于生成只读摘要或打印视图。
- 附件：可与整段子模块绑定（如「管理部门证明文件」），支持**下载模板 + 上传**，并约束格式（如 PDF、最大 1MB）。

**参考界面特征**（与现有系统截图一致）：同一块内先出现汇总句式 + 保存，再出现单独一行的证明上传。

### 3.2 List 类型（列表 / 明细表）

**适用**：课程清单、荣誉列表等**可增删改排序**的多条记录。

**能力要点**：

- **列定义**：列名、数据类型（文本、数字、布尔、枚举、日期、附件等）、是否可编辑、列宽与展示顺序。
- **行操作**：添加、修改、删除、排序（与业务规则一致，如最多 N 行）。
- **行级附件**：某列类型为「证明文件」，与该行数据绑定存储。

**参考界面特征**：表上方工具栏「添加 / 修改 / 删除 / 排序」，表格含勾选、序号及业务列。

### 3.3 组合方式

- 同一 **模块** 下可并列多个子模块：例如「本科教学开设课程」= Map（汇总句）+ 附件 + List（明细表）；其下再并列「研究生指导开设课程」重复类似结构。
- **子模块顺序**由配置决定，用于前端渲染与导出目录顺序。

## 4. 配置内容要素（建议字段）

以下为逻辑模型，实现时可落为 JSON Schema、数据库表或 YAML，按团队规范选型。

### 4.1 子模块公共属性

| 逻辑项 | 说明 |
|--------|------|
| `id` / `key` | 稳定标识，用于存储路径与接口 |
| `title` | 展示标题 |
| `type` | `map` \| `list` |
| `helpText` | 顶部说明（可为富文本或 Markdown） |
| `order` | 排序 |

### 4.2 Map 特有

| 逻辑项 | 说明 |
|--------|------|
| `fields[]` | 字段定义：name、label、widget、validation、默认值 |
| `sentenceTemplate` | 可选，句式模板与占位符 |
| `attachments[]` | 见下节 |

### 4.3 List 特有

| 逻辑项 | 说明 |
|--------|------|
| `columns[]` | 列定义：name、title、cellType、options、width |
| `maxRows` | 如 10 |
| `rowAttachments` | 若附件在列中定义，则列类型为 `file` 即可 |
| `toolbar` | 是否启用添加/编辑/删除/排序及权限 |

### 4.4 附件定义（共用）

| 逻辑项 | 说明 |
|--------|------|
| `key` | 存储键 |
| `label` | 文案（可含格式说明） |
| `required` | 是否必填 |
| `accept` | 如 `.pdf` |
| `maxSize` | 如 1MB、2MB |
| `templateUrl` | 可选，「下载模板」 |

---

## 5. 数据存储与实例数据（原则）

- **配置**：按「项目 + 版本」存储，发布后对申报人只读；管理员可升版。
- **实例数据**：建议路径形如  
  `projectId / moduleKey / subModuleKey`  
  - Map：`{ fieldKey: value, attachments: { key: fileRef } }`  
  - List：`rows[]`，每行结构与 `columns` 一致，附件存文件引用。
- **文件**：统一走对象存储或现有「用户目录 + URL」方案，配置中只保存**业务键 → 文件元数据**映射。

## 6. 与截图示例的对应关系

### 6.1 「立德树人」— 概述 Tab

- **模块**：立德树人 → **子模块「概述」**  
  - Map：上传说明类字段 + 可选专家评价附件。  
  - 体现：**固定字段 + 多附件 + 模板说明**。

### 6.2 「人才培养」Tab

- **模块**：人才培养  
  - **子模块「本科教学开设课程」**：  
    - Map：近五年讲授课程门数、学时、人次等（句式汇总）。  
    - 附件：本科教学管理部门证明（PDF，≤1M，可配模板）。  
    - List：课程明细表（列：课程名称、学期数、总学时、人次、是否核心/精品、证明文件等），行数上限与操作按钮由配置表达。  
  - **子模块「研究生指导开设课程」**：结构可复用另一套 `subModule` 配置。

该划分说明：**同一 Tab（模块）内多个子模块串联**，每个子模块内部再组合 Map 与 List。

## 7. 前端渲染与校验

- **配置加载**：进入申报页时拉取当前项目生效配置 → 按模块/子模块动态挂载路由或折叠面板。
- **Map**：由字段定义生成 Form；若有 `sentenceTemplate`，可双向绑定或拆成独立字段再在预览区拼句。
- **List**：由 `columns` 生成 Table + 行内编辑或弹窗编辑；校验行数与单元格规则。
- **上传**：统一组件，读取 `accept`、`maxSize`、`templateUrl`；标签与 `htmlFor` 等行为按无障碍与防误触规范处理（参见基本信息页面对 Upload 的实践）。

---

## 8. 权限与流程（简述）

- 配置管理：仅管理员或项目管理员可编辑、发布配置版本。
- 申报人：仅能填写当前开放项目下的实例数据。
- 审批/导出：可按模块与子模块聚合生成 PDF 或打包附件（实现阶段再细化）。

---

## 9. 扩展与约束

| 方向 | 说明 |
|------|------|
| 新项目 | 新建项目 + 新配置版本，复用通用子模块模板（如标准 List 课程表）。 |
| 字段变更 | 通过版本号区分；旧数据对新增字段可空或迁移脚本。 |
| 国际化 | `title`/`label` 可扩展为 i18n key。 |
| 性能 | List 行数上限、分页或虚拟滚动在实现层约束。 |

---

## 10. 后续工作（实现向）

1. 将本模型落实为 **JSON Schema 或数据库表结构**（`project`、`module`、`sub_module`、`field`、`attachment` 等）。
2. 提供**管理端**配置界面或「导入 YAML/JSON」能力。
3. 申报端 **配置驱动表单引擎** 与现有「分模块保存」接口对齐（参考当前 `declaration_basic` 等模块拆分方式）。
