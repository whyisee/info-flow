# 用户权限管理设计方案

> 面向「信息流转」类业务系统，在现有「固定角色枚举」之上，给出可演进至中大型组织使用的权限与组织模型设计。  
> 本文档为**方案级**说明，不涉及具体表字段的最终定稿，实施时需结合现有库表与迭代节奏裁剪。

---

## 1. 背景与目标

### 1.1 现状简述

当前实现常见形态为：

- 用户表上挂载**单一角色字段**（如 `teacher` / `dept_admin` / `school_admin` / `expert`）；
- 前后端按角色**硬编码**菜单可见性与接口校验；
- **数据范围**（只能看本部门、本院系等）多在业务代码里写死或缺失。

该方式在原型与小型试点阶段成本低，但难以支撑：

- 同一用户多角色、兼职岗位；
- 角色权限微调（仅开放部分菜单或接口）；
- 按组织层级、项目、材料类型的**行级数据隔离**；
- 审计与合规要求的「谁、在何时、对何资源、做了何操作」。

### 1.2 设计目标

| 维度 | 目标 |
|------|------|
| **功能权限** | 菜单、路由、按钮、API 可配置、可组合，避免处处改代码 |
| **数据权限** | 支持本人、本部门、本条线、全校/全局等可扩展策略 |
| **组织与身份** | 支持部门、岗位、用户组；用户与角色/岗位可一对多 |
| **可演进** | 可分阶段落地，与现有角色模型兼容迁移 |
| **可审计** | 关键操作留痕，权限变更可追溯 |

---

## 2. 设计原则

1. **默认拒绝（Deny by default）**：未显式授予的权限一律不可访问。  
2. **权限与业务解耦**：权限元数据（资源、操作）与业务表分离，业务只引用「资源标识」与「策略结果」。  
3. **数据权限独立建模**：功能权限（能否进菜单）与数据权限（列表里能看哪些行）分开判定，避免混在一个 `role` 字符串里。  
4. **优先 RBAC，预留 ABAC**：以**基于角色的访问控制（RBAC）**为主；属性（部门、项目、材料状态等）通过**数据范围策略**或未来 **ABAC** 扩展。  
5. **前后端一致**：前端隐藏无权限入口，后端**必须**再次校验，不信任前端。

---

## 3. 核心概念模型

### 3.1 主体（Subject）

- **用户（User）**：登录主体，唯一账号。  
- **可选扩展**：用户组（Group）、服务账号、API 密钥（若对外开放接口）。

### 3.2 资源（Resource）

逻辑上的受保护对象，建议采用**统一资源标识**：

- **格式示例**：`domain:resource[:instance]`  
  - 如：`declaration:project`、`declaration:material`、`survey:form`、`system:user`。  
- **菜单/路由**可映射为资源：`ui:menu:declaration.projects`。  
- **API** 可映射为资源 + 操作：`declaration:material` + `read|create|update|submit|approve`。

### 3.3 操作（Action）

在资源上的动词，建议有限集合，便于配置与审计：

| 类别 | 示例 |
|------|------|
| 通用 | `read`、`create`、`update`、`delete`、`export` |
| 流程 | `submit`、`withdraw`、`approve`、`reject`、`return` |
| 管理 | `assign_role`、`publish`、`close` |

**权限点** = 资源 + 操作（及可选数据条件，见数据权限）。

### 3.4 权限（Permission）

一条可授予的**原子能力**，例如：

- `declaration:project:create`
- `declaration:material:approve`
- `system:user:read`

系统预置权限清单（Permission Catalog），版本化维护，新增业务时**追加**权限定义而非改角色枚举。

### 3.5 角色（Role）

- **角色**是权限的集合，面向**岗位或职责**命名（如「院系教务」「校级申报管理员」）。  
- 支持 **RBAC1**（角色继承）：子角色继承父角色权限（可选，二期）。  
- **用户—角色**：多对多；可设**主角色**用于默认展示与简化 UI。  
- 与现有「单角色字段」兼容方案：迁移期保留 `role` 作为**主角色**或**默认角色模板**，同步到 `user_role` 表。

### 3.6 数据权限 / 数据范围（Data Scope）

解决「同一菜单进来，列表数据不同」的问题。常见策略：

| 策略编码 | 含义 | 典型场景 |
|----------|------|----------|
| `SELF` | 仅本人数据 | 教师看自己的申报 |
| `DEPT` | 本部门/所在单位 | 部门管理员 |
| `DEPT_TREE` | 本部门及下级组织 | 多级院系 |
| `PROJECT` | 绑定项目范围 | 项目协管员 |
| `ALL` | 全校/全局 | 校级管理员 |
| `CUSTOM` | 自定义规则集 | 按项目成员表、标签等 |

实现上可为角色或权限绑定**默认数据范围**；特殊接口再通过**策略表达式**覆盖（见下文扩展）。

## 4. 功能权限体系

### 4.1 分层

1. **菜单 / 路由权限**：控制侧栏、顶部模块、页面是否可见；可与 `ui:menu:*` 资源对齐。  
2. **界面元素**：按钮、批量操作、导出、敏感字段脱敏（前端控制展示，后端仍校验）。  
3. **API 权限**：每个受保护接口绑定所需 `permission` 列表或 `resource:action`；网关或中间件统一鉴权。

### 4.2 与「申报 / 问卷 / 系统」模块的关系

- **顶部模块**可作为**一级资源域**：`declaration`、`survey`、`system`。  
- 侧栏二级菜单对应**二级资源**或 `ui:menu` 节点。  
- 权限配置界面可按**树形**展示，与当前产品信息架构一致，降低运营理解成本。

### 4.3 权限缓存与失效

- 用户登录后下发 **权限列表或 JWT 内嵌权限摘要**（注意体积）；或会话内服务端缓存 `userId -> permissions`。  
- 角色/权限变更后需**失效会话或推送刷新**，避免长时间不一致。

---

## 5. 数据权限体系

### 5.1 判定顺序（建议）

对每个「带数据的读/写」请求：

1. **功能权限**：是否具备 `resource:action`。  
2. **数据范围**：在查询层附加 `WHERE` 条件或 ORM 过滤（部门、创建人、项目成员等）。  
3. **行级特例**：单条 `GET` 时再校验是否落在范围内，防止改 ID 越权。

### 5.2 组织维度

- **组织单元（OrgUnit）**：树形结构（学校—学院—系），用户挂靠多个组织时可带**主组织**。  
- **数据归属**：材料、项目、问卷等表增加 `org_id` / `dept_id` / `owner_user_id` 等，便于范围过滤。

### 5.3 与审批流的关系

- 「当前环节审批人」往往来自**动态指派**（项目、材料状态），不仅是静态角色。  
- 建议：**功能权限**控制「能否进入审批工作台」；**数据权限 + 业务状态机**决定「待办列表是否出现该条」。  
- 避免把「待办」完全绑在静态角色上，否则无法支持加签、代理审批等扩展。

---

## 6. 管理功能设计（权限运营）

### 6.1 角色管理

- 角色的创建、编辑、停用（停用后不再分配新用户，历史会话按策略失效）。  
- 角色绑定**权限集合**（多选权限树）。  
- 可选：**角色复制**、从模板初始化（如「校级管理员模板」）。

### 6.2 用户与授权

- 用户分配**多个角色**；可设生效时间（用于临时授权、项目周期）。  
- **用户组**：批量授权（如「2025 申报秘书组」），减少重复操作。

### 6.3 权限定义维护

- 仅超级管理员或「系统配置」角色可改权限元数据（避免业务管理员误改系统级权限）。  
- 新功能上线时**登记权限点**（CI 或文档门禁），防止遗漏后端校验。

### 6.4 审计

- 记录：权限变更、角色变更、用户角色绑定变更、敏感数据导出。  
- 字段建议：操作者、对象类型、对象 ID、前后快照、时间、来源 IP/UA。

---

## 7. 技术落地要点（与现有栈对齐）

### 7.1 后端（如 FastAPI）

- **依赖注入**：`get_current_user` → 加载权限列表 → `require_permission("declaration:material:approve")`。  
- **统一异常**：403 与 404（资源不存在或对当前用户不可见时是否返回 404 防枚举，需产品决策）。  
- **查询封装**：Repository 层接收 `user + data_scope` 自动拼接条件。

### 7.2 前端（如 React）

- **路由守卫**：无菜单权限则重定向或 403 页。  
- **指令/组件**：`<Can permission="...">` 控制按钮。  
- **菜单生成**：由后端返回「当前用户可见菜单树」或前端根据权限列表过滤静态配置。

### 7.3 数据迁移

- 保留现有 `role` 字段，编写**种子脚本**：为每个旧角色生成等价 `Role` + `Permission` 绑定。  
- 新用户直接走新表；老数据逐步迁移到 `user_role` 关联表。

## 8. 扩展方向（非一期必做）

| 方向 | 说明 |
|------|------|
| **岗位（Position）** | 与人事系统对齐，一人多岗，岗变权限变 |
| **委托 / 代理** | 审批代理、时间区间生效 |
| **ABAC** | 基于标签、密级、项目属性的动态策略引擎 |
| **多租户** | `tenant_id` 隔离，权限域按租户裁剪 |

---

## 9. 分期实施建议

| 阶段 | 内容 | 价值 |
|------|------|------|
| **一期** | 权限点目录 + 角色表 + 用户多角色 + 后端接口统一鉴权；菜单由权限驱动 | 去掉硬编码角色分支的主要痛点 |
| **二期** | 数据范围策略 + 组织树 + 列表/详情过滤 | 支撑多部门、校级/院级差异 |
| **三期** | 审计中心、用户组、角色继承、代理审批 | 合规与运维效率 |

---

## 10. 小结

本方案将**身份（用户）**、**授权（角色—权限）**、**资源与操作**、**数据范围**分层设计，在不过早引入过重引擎的前提下，为复杂组织、多业务线（申报、问卷、系统管理）预留统一扩展面。实施时建议先冻结**权限命名规范**与**接口鉴权规范**，再逐步替换现有 `if role == school_admin` 式逻辑，以降低回归风险。
