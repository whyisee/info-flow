# 项目状态 / 申请(材料)状态 / 审批状态 / 用户与资料状态梳理（以代码为准）

本文档用于梳理项目内“状态”字段的取值、含义、流转与前端展示映射，避免不同页面/接口对同一字段的解释不一致。

---

## 1. 项目（ApplyProject）状态

### 1.1 字段
- **表**：`backend/app/models/project.py` → `ApplyProject.status: int`
- **含义**：项目生命周期（是否可申报）

### 1.2 取值枚举
- **0**：草稿（draft）
- **1**：已发布（published）
- **2**：已关闭（closed）

代码来源：
- 后端模型注释：`backend/app/models/project.py`
- 前端展示：`frontend/src/utils/constants.ts` → `PROJECT_STATUS`

### 1.3 关键业务规则
- **教师端只能看到已发布项目**：`backend/app/api/projects.py` 的 `list_projects` 对 teacher 加了 `ApplyProject.status == 1` 过滤。

## 2. 申报材料/申请（ApplyMaterial）状态（全局流程状态）

### 2.1 字段
- **表**：`backend/app/models/material.py` → `ApplyMaterial.status: int`
- **含义**：材料整体所处审批流程阶段（草稿/第k环节/办结/驳回）

### 2.2 取值枚举（可变环节数）
> 这里的 \(n\) 是环节数：  
> - 若 `approval_snapshot` 存在：\(n = len(approval_snapshot.steps)\)  
> - 若 `approval_snapshot` 为空（legacy）：\(n = 3\)

- **0**：草稿（draft）
- **1..n**：第 \(k\) 环节进行中（`status = k`）
- **n+1**：已通过 / 办结（done）
- **5**：已驳回（rejected）

代码来源：
- 后端模型注释：`backend/app/models/material.py`
- 后端常量：`backend/app/services/approval_flow_service.py`  
  - `LEGACY_STEP_COUNT = 3`  
  - `REJECTED_STATUS = 5`  
  - `done_status_value(material) = step_count(material) + 1`
- 前端展示：`frontend/src/utils/materialApproval.ts`  
  - `materialStepCount()`  
  - `materialStatusLabel(status, stepCount)`

### 2.3 关键流转点（后端）
主要由 `backend/app/api/approvals.py` 修改：
- **提交材料**：`backend/app/api/materials.py` 的 `submit_material` 将 `material.status = 1` 并写入 `submitted_at`（同时把项目审批流快照复制到 `approval_snapshot`）。
- **通过**：`POST /approvals/{material_id}/approve`  
  - 写入 `ApproveRecord(status=1)`  
  - 将 `material.status` 推进到 `next_status`（由角色/流程决定）
- **退回**：`POST /approvals/{material_id}/return`  
  - 写入 `ApproveRecord(status=2)`  
  - 将 `material.status = 0` 且 `approval_snapshot = None`（回到草稿并清空快照）
- **驳回**：`POST /approvals/{material_id}/reject`  
  - 写入 `ApproveRecord(status=3)`  
  - 将 `material.status = REJECTED_STATUS(5)`

> 注意：`status=5` 是“驳回”的全局状态值；而审批记录里的 `ApproveRecord.status=3` 才是“某次操作是驳回”。

## 3. 审批记录（ApproveRecord）状态（单次操作结果）

### 3.1 字段
- **表**：`backend/app/models/approval.py` → `ApproveRecord.status: int`
- **含义**：某个审批人对某材料的一次动作结果

### 3.2 取值枚举
- **1**：通过（approved）
- **2**：退回（returned）
- **3**：驳回（rejected）

代码来源：
- 后端模型注释：`backend/app/models/approval.py`
- 前端常量：`frontend/src/utils/constants.ts` → `APPROVAL_STATUS`

### 3.3 关联字段
- `ApproveRecord.step_index`：会签“通过”时记录所属环节 0..n-1（退回/驳回可为空）
- `ApproveRecord.lane_index`：并行块内轨道 0..k-1；非并行环节为 null

## 4. “待我审批”列表的状态（我的处理状态）

### 4.1 字段（接口返回）
- **接口**：`GET /approvals/queue`
- **返回模型**：`backend/app/schemas/approval.py` → `ApprovalQueueOut.my_action_status: int`
- **前端类型**：`frontend/src/types/index.ts` → `ApprovalRecord.my_action_status?: number`

### 4.2 取值枚举
- **0**：未处理（仍需我处理当前环节/并行子轨）
- **1**：通过（我最新一条 `ApproveRecord.status == 1`）
- **2**：驳回（我最新一条 `ApproveRecord.status in (2,3)`，即退回或驳回都归为“驳回”口径）

代码来源：
- 后端：`backend/app/api/approvals.py` → `list_my_queue`
- 前端列表：`frontend/src/pages/approvals/ApprovalCenterPage.tsx`（筛选与展示）

> 说明：这里的“驳回”是**列表口径**，把“退回/驳回”都统一归为“驳回”，用于“我的处理状态”三态展示与筛选。

## 5. 用户账号状态（User.status）

### 5.1 字段

- **含义**：账号是否可登录/是否逻辑删除

### 5.2 取值枚举
- **active**：正常

代码来源：

- 用户筛选：`backend/app/api/users.py`（`user_status` query：active/deleted/all）

## 6. 用户资料/模块配置（UserModuleConfig）状态与“资料提交状态”

### 6.1 模块配置行状态（UserModuleConfig.status）
- **表**：`backend/app/models/user_module_config.py` → `UserModuleConfig.status: str`
- **默认值**：`active`
- **当前使用情况**：代码中仅存储/回显（`backend/app/api/module_config.py`），未看到复杂流转与枚举约束。

### 6.2 资料“提交状态”（form_status）
> 这是**业务状态**，不在独立列里，而是放在 `declaration_basic.config` 里。

- **字段**：`form_status`（前端常量：`frontend/src/pages/declaration/profile/profileModuleFields.ts` → `FORM_STATUS_KEY`）
- **取值**：
  - `draft`
  - `submitted`
- **存储位置**：用户模块配置 `UserModuleConfig(module="declaration_basic").config.form_status`

> 注意：此状态与 `ApplyMaterial.status`（材料审批状态）不是同一个维度；前者是“个人资料/模块配置的提交标记”，后者是“某个项目申报材料的审批流状态”。

## 8. 容易混淆的点（建议统一口径）

- **ApplyMaterial.status=5** 是“材料被驳回（终态）”，但 **ApproveRecord.status=3** 才是“某次操作是驳回”。
- “待我审批列表”的 **my_action_status=2** 当前把 **退回(2) + 驳回(3)** 合并成“驳回”口径；若未来要拆成“退回/驳回”需要扩展枚举与前端筛选。
- `form_status` 属于“个人资料模块提交状态”，与材料审批状态是不同对象/不同生命周期。
